# MacOS Native Containers vs Docker

Apple's `container` tool and Containerization framework (announced WWDC 2025, Swift-based, Apache 2.0) vs Docker Desktop on Mac.

## Similarities
- Both run Linux containers on macOS through a VM, since macOS has no Linux kernel of its own.
- Both use OCI-compatible images, so an image built with one runs on the other (and on containerd, Podman).
- Both sit on top of Apple's virtualization APIs (Hypervisor.framework / Virtualization.framework) at some layer.

## Differences
- **VM model**: Docker Desktop runs all containers inside one shared Linux VM. Apple runs one lightweight VM per container.
- **Isolation**: Apple's per-container VM shrinks the shared-kernel attack surface. Docker containers share the single VM's kernel via namespaces and cgroups.
- **Networking**: Apple gives each container its own IP (192.168.64.x range), cutting the need for port forwarding. Full container-to-container networking firms up in macOS 26 Tahoe.
- **Startup**: Despite a VM per container, Apple boots in under a second via a minimal kernel and the vminitd init system.
- **Ecosystem**: Docker has Compose, a desktop UI, debugging tools, and Hub. Apple's runtime is lower level and currently has no Compose equivalent.
- **Platform fit**: Apple Container targets Apple Silicon and works best on macOS 26 Tahoe. Networking is limited on macOS 15 and it is not built for Intel Macs.

## Takeaway
Not a drop-in Docker replacement, but a strong Mac-first option where security isolation and Apple Silicon performance matter more than orchestration tooling. The two can also coexist: Docker can lean on the native stack and drop its own background VM.
