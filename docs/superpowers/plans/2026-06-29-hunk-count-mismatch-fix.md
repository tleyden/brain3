# Plan: Fix Hunk Count Mismatch in Patch Generation

## RCA Validation

The RCA is **correct**. The failure is 100% client-side (patch generator), not in Brain3's application logic.

Evidence from `patch.py:84-87`:

```python
actual_old_count = sum(1 for op, _ in hunk_lines if op in {" ", "-"})
actual_new_count = sum(1 for op, _ in hunk_lines if op in {" ", "+"})
if actual_old_count != old_count or actual_new_count != new_count:
    raise PatchError("invalid_patch", "Hunk line counts do not match header counts")
```

This has always been there. The server already validates counts and produces the exact error seen. The patch that was generated had:

- Header: `@@ -117,3 +117,3 @@` (claims old=3, new=3)
- Body: 1 deletion + 1 insertion (actual old=1, new=1)

The discrepancy: the generator picked a 3-line window, wrote the 3-line header, then emitted only the changed line(s) — dropping the context lines entirely.

All 21 tests pass.

## Root Cause: Where the Bad Patch Comes From

The patch is generated by the LLM (Claude or ChatGPT) calling `vault_apply_unified_diff`. The MCP tool description tells callers what the patch format is, but does not explicitly state that the `N` in `@@ -L,N +L,N @@` must equal the exact number of lines (context + changed) that follow in the body. LLMs tend to copy the `n=3` context window from `difflib.unified_diff` semantics without verifying that the context lines are actually present.

## Fix Options

### Option A — Improve the tool description (lowest risk, recommended first)

Add an explicit rule to the `vault_apply_unified_diff` MCP tool docstring:

> **The count `N` in `@@ -L,N +L,N @@` must equal the number of lines actually present in the hunk body.** Context lines (`<SPACE>`) count toward both old and new. If you include no context, use `@@ -L,1 +L,1 @@`. If you include one context line before and after, use `@@ -L,3 +L,3 @@`. Do not copy a count from `git diff` output unless you also copy the context lines that go with it.

Add a minimal and a contextual example directly in the docstring so the model can pattern-match.

**File:** `brain3-mcp-vault-tools/src/brain3_mcp_vault_tools/tools/patch.py` — the docstring of `vault_apply_unified_diff`.

### Option B — Add a pre-flight assertion in the patch generator (defensive)

Suggested by the RCA. Since the generator is the LLM, this has no direct place in the codebase. However, we could add a **client-side helper** or example script that validates before sending:

```python
def _validate_hunk_counts(header_old, header_new, body_lines):
    old = sum(1 for l in body_lines if l[0] in {' ', '-'})
    new = sum(1 for l in body_lines if l[0] in {' ', '+'})
    assert old == header_old and new == header_new
```

This could live in a future `brain3_mcp_vault_tools.patch_builder` helper if we ever want to offer a diff-generation utility.

### Option C — Add a server-side generate-diff tool (most robust, most work)

Expose a new MCP tool `vault_generate_diff(path, new_content)` that takes the full intended content, diffs it internally using `difflib.unified_diff`, and returns a valid patch string or applies it directly. This moves patch generation to the server where it can never be wrong.

**Not recommended now** — changes the API surface and AGENTS.MD says to start with smaller steps.

## Recommended sequence

1. **Done:** Tests in `test_tool_write_patch_api.py` confirming the server correctly rejects and accepts all count combinations.
2. **Next:** Update the `vault_apply_unified_diff` docstring (Option A) with the count rule and examples.
3. **Later if A is insufficient:** Consider Option C (server-side diff generation).
