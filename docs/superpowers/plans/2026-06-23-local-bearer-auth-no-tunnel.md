# Plan: Local MCP Port + Env-var Upstream Secret

**Date**: 2026-06-23  
**Status**: Proposed

## Motivation

Requiring Cloudflare tunnels and full OAuth2.1 is too complex for local-only use cases. Claude Code (and other local clients) should be able to reach the gateway on a localhost-only port with a simple static bearer token — no tunnel, no OAuth dance, no domain setup.

The gateway remains the sole orchestrator: clients never connect directly to containers. This matters because the gateway already knows how to route to multiple containers, handle health checks, and proxy with the upstream shared secret. Bypassing it would leak that complexity to clients.

A separate security improvement: the upstream shared secret (gateway → container) is currently written to `/tmp`, readable by any local process. It should never touch disk.

---

## Architecture: Two Listeners, One Gateway

The gateway runs two Axum listeners bound at startup:

| Listener | Bind address | Auth | Routes |
|----------|-------------|------|--------|
| **Public port** | `127.0.0.1:<B3_OAUTH2_GATEWAY_PORT>` (default 8421) | OAuth2.1 bearer tokens | All existing routes (OAuth, MCP, health) |
| **Local port** | `127.0.0.1:<B3_LOCAL_MCP_PORT>` (default 8422) | Static bearer token | `/mcp`, `/mcp/`, `/mcp/{*path}`, `/health` only |

The local port is **localhost-only by construction** — it binds `127.0.0.1`, never `0.0.0.0`. A CF tunnel or any other external ingress only points at the public port. Local clients (Claude Code, Claude Desktop) use the local port directly.

Both listeners share the same `AppState` and proxy to the same upstream container(s). The routing logic is identical; only the auth validation differs.

## Part 1 — Local MCP Port

### 1a. Setup TUI prompt

During first-run setup (`apps/gateway/src/setup_tui.rs` / `crates/core/src/application/first_run_setup.rs`), add a question:

> **Enable local MCP access?**
> Runs a localhost-only port so Claude Code and other local clients can connect without OAuth.
> \[Y/n\] (default: yes)

If yes, write to `.env`:
```
B3_LOCAL_MCP_PORT=8422
LOCAL_GATEWAY_MCP_REVERSE_PROXY_BEARER_TOKEN=<auto-generated 32-byte hex>
```

If no, neither var is written and the local port is not started.

### 1b. `crates/core/src/domain/model.rs`

Add `LocalMcpConfig` and attach it to `GatewayConfig`:

```rust
#[derive(Debug, Clone)]
pub struct LocalMcpConfig {
    pub port: u16,
    pub bearer_token: String,
}
```

```rust
pub struct GatewayConfig {
    // ... existing fields ...
    pub local_mcp: Option<LocalMcpConfig>,  // None = disabled
}
```

`OAuthConfig` and the public port are unchanged.

### 1c. `crates/platform/src/config/env_file.rs`

Read `B3_LOCAL_MCP_PORT` and `LOCAL_GATEWAY_MCP_REVERSE_PROXY_BEARER_TOKEN`:
- If `B3_LOCAL_MCP_PORT` is set → parse port, require `LOCAL_GATEWAY_MCP_REVERSE_PROXY_BEARER_TOKEN` to be non-empty.
- If `LOCAL_GATEWAY_MCP_REVERSE_PROXY_BEARER_TOKEN` is set but `B3_LOCAL_MCP_PORT` is absent → use default port 8422.
- If neither is set → `local_mcp: None` (local port disabled).
- Never auto-generate the token at runtime; setup TUI generates it once and persists it to `.env`.

Also: **change `B3_CF_QUICK_TUNNEL` default from `true` → `false`**. Tunnel is opt-in only.

### 1d. `crates/platform/src/http/router.rs`

Add a second router builder for the local port:

```rust
pub fn build_local_router<P: McpProxyPort + 'static>(
    state: AppState<P>,
    bearer_token: String,
) -> Router {
    Router::new()
        .route("/health", get(health))
        .route("/mcp",        get(local_mcp_proxy::<P>) .post(local_mcp_proxy::<P>).delete(local_mcp_proxy::<P>))
        .route("/mcp/",       get(local_mcp_proxy::<P>) .post(local_mcp_proxy::<P>).delete(local_mcp_proxy::<P>))
        .route("/mcp/{*path}", get(local_mcp_proxy::<P>).post(local_mcp_proxy::<P>).delete(local_mcp_proxy::<P>))
        .fallback(fallback)
        .layer(TraceLayer::new_for_http())
        .with_state(state)
}
```

`local_mcp_proxy` validates the static bearer token before delegating to the same `ProxyMcpUseCase` as the public port.

### 1e. `crates/platform/src/http/mcp_handlers.rs`

Add `local_mcp_proxy` handler:
- Extracts `Authorization: Bearer <token>` header.
- **Constant-time compare** against `config.local_mcp.bearer_token` (use `subtle` crate or manual byte-by-byte to avoid timing leaks).
- On success: same proxy logic as `mcp_reverse_proxy`.
- On failure: `401` with a plain JSON error — no `WWW-Authenticate: Bearer resource_metadata=...` hint (that's OAuth-specific).

The existing `mcp_reverse_proxy` and its OAuth token validation are untouched.

### 1f. `apps/gateway/src/server.rs`

Spawn a second `tokio::spawn` for the local listener when `config.local_mcp.is_some()`:

```rust
if let Some(local_cfg) = &config.local_mcp {
    let local_addr = format!("127.0.0.1:{}", local_cfg.port);
    let local_router = build_local_router(state.clone(), local_cfg.bearer_token.clone());
    tokio::spawn(async move {
        axum::serve(TcpListener::bind(local_addr).await?, local_router).await
    });
}
```

### 1g. `apps/gateway/src/tui/screens.rs`

Add a "Local MCP" section to the runtime screen when local port is active:

```
Local MCP
  Endpoint : http://localhost:8422/mcp
  Token    : abc1234...   ← configure in .mcp.json
```

## Part 2 — Upstream Secret via Env Var (gateway → container)

**Problem**: Secret is written to `/tmp/brain3-mcp-upstream-secret` and bind-mounted into the container. Any local process can read `/tmp`.

**Fix**: Generate secret in memory → inject directly as `B3_UPSTREAM_SHARED_SECRET` env var into the container at startup. Never write to disk.

### Changes

**2a. `crates/core/src/domain/model.rs`**

- `MCPReverseProxyConfig`: replace `upstream_secret_file: PathBuf` with `upstream_secret: String`
- `ContainerStartupConfig`: replace `upstream_secret_dir: PathBuf` with `upstream_secret: String`

**2b. `crates/platform/src/config/env_file.rs`**

- Remove `B3_OAUTH2_GATEWAY_UPSTREAM_SECRET_FILE` env var and parsing.
- Generate secret inline in `load()` using the same randomness as before (64-char alphanumeric), storing it in `GatewayConfig` fields in memory only.
- If `B3_UPSTREAM_SHARED_SECRET` env var is set on the host, use that value directly (allows power users to pin it across restarts without a file).
- Never write the secret to disk.

**2c. `crates/platform/src/config/upstream_secret.rs`**

Delete this file. Logic moves inline to `env_file.rs`.

**2d. `crates/platform/src/container/startup.rs`**

- Remove the `upstream_secret_dir` bind-mount (`/run/brain3` → `/run/brain3`).
- Add `("B3_UPSTREAM_SHARED_SECRET".into(), startup.upstream_secret.clone())` to the env vars injected into the container.
- Remove `B3_UPSTREAM_SHARED_SECRET_FILE` env var from container env (no longer needed).

**2e. `crates/platform/src/runtime/bootstrap.rs`**

`upstream_secret` now comes from `config.mcp_reverse_proxy.upstream_secret` (in-memory field), not from `upstream_secret::read_or_create()`.

**2f. `brain3-mcp-vault-tools/src/brain3_mcp_vault_tools/config.py`**

Read upstream secret with priority order:
1. `B3_UPSTREAM_SHARED_SECRET` env var (direct value — new primary path)
2. `B3_UPSTREAM_SHARED_SECRET_FILE` (file path — backward compat)
3. Fail loudly at startup if neither is set.

---

## What stays the same

- OAuth2.1 mode on the public port is completely unchanged — no regressions for remote/CF tunnel users.
- Container isolation strategies, network modes, health probe logic — unchanged.
- The `x-brain3-upstream-secret` header on gateway→container proxied requests — unchanged.
- The gateway is the sole orchestrator; clients never communicate with containers directly.

---

## End-state: Claude Code MCP config

Users add to `.mcp.json`:

```json
{
  "brain3": {
    "type": "http",
    "url": "http://localhost:8422/mcp",
    "headers": { "Authorization": "Bearer <token-from-env>" }
  }
}
```

No tunnel. No OAuth. No domain. Gateway still routes, orchestrates, and proxies everything.

---

## Implementation Order

1. **Part 2 first** (upstream secret → env var): pure security improvement, no UX impact, self-contained.
2. **Part 1 second** (local MCP port): touches `model.rs`, `config/env_file.rs`, `server.rs`, router, handlers, TUI.
